# Local Inference Stack

中文名称：本地模型推理与运营栈。

仓库名：`local-inference-stack`。

## 项目定位

本项目面向单机消费级 GPU，负责把本地模型制品转化为可重复部署、可度量、可验收、
可回滚并可长期运营的推理服务。它不是模型训练仓库、Agent 应用或 ModelPort 的副本。

核心目标是：

```text
模型制品 -> 推理运行时 -> 协议网关 -> 验收基线 -> 实时观测 -> 持续优化
```

对全新 checkout，目标闭环进一步明确为：Agent 先只读识别主机能力，从受审查的
Catalog 给出带证据等级的候选；用户批准后才下载、校验和部署，并用无密钥直连门禁
确认可用。不同硬件不复制脚本，而是物化不同模型 Profile。

## 分层边界

| 层次 | 本项目负责 | 不负责 |
| --- | --- | --- |
| 制品与运行时 | 模型校验、容器、GPU/KV/上下文、Profile、端口和回滚 | 训练与微调 |
| 性能工程 | 预填充、解码、缓存、并发、显存和单变量 A/B | 无验收依据的参数追新 |
| 协议集成 | 与 ModelPort 的运行时能力契约和真实上游验收 | 认证、配额和账本实现 |
| 质量保障 | Reasoning、长上下文、Token、Tool Use 和回归基线 | 业务 Prompt 质量 |
| 运营观测 | 聚合报告、实时运行台、告警、版本和部署记录 | 保存 Prompt、回复或工具参数 |

ModelPort 负责 Anthropic/OpenAI 边界、认证、路由、配额、账本和通用 Tool Use
适配；上层应用负责 Prompt、RAG、Agent、业务工具和用户体验。

## 命名模型

项目名称保持通用，具体模型和硬件进入部署档案：

```text
项目：Local Inference Stack
部署：Qwen3.5-9B / RTX 5070 Ti / 128K
```

服务名、容器名和网络别名属于部署契约，不因项目改名而随意变更。当前稳定契约是
`qwen35-9b-q5km`、`qwen-runtime`、`local_qwen` 和宿主机诊断端口 `18080`。
机器可读的跨项目能力边界位于 `contracts/local-qwen-provider-v1.json`。

## 演进顺序

### P0：运营闭环（已完成）

1. 运行台与报告显式区分当前部署和 ModelPort 全局统计口径。
2. 统一 quick/standard/full 验收入口，并输出版本化结果。
3. 为长期趋势增加有保留上限的聚合时序数据。
4. 固化本项目与 ModelPort 的 Provider capability 契约。

### P1：性能与质量（已完成基础闭环）

1. 增加响应开始延迟、缓存命中、上下文长度和运行时 Reasoning 状态；真正的流式
   TTFT 与 Reasoning/正文 Token 拆分列入下一阶段。
2. 建立代码、推理、结构化输出和 Tool Use 固定合成质量集；长文检索由 92K/118K
   专项验收覆盖。
3. 建立模型、镜像、配置和验收结果组成的发布清单。
4. 继续把通用适配沉淀到 ModelPort capability，避免模型专用分支。

当前已落地精确上下文准入、Tool Use 终态、版本漂移检查、验收证据、分层时序历史、
质量集和独立候选发布入口。后续 P1 工作主要是积累真实失败模式并扩展合成回归，不再
以增加无证据的运行参数为目标。

### P1.1：推理与 Tool Use 可靠性（已规划）

1. ModelPort 对模型生成的工具参数执行完整 JSON Schema 校验。
2. 将请求成功、模型调用、Schema 通过、Tool Result 续接和最终任务成功拆成独立指标。
3. 用 40--60 个闭环 Case 覆盖多工具、多步、错误恢复、流式与注入场景。
4. 使用验证器驱动的 `fast -> code -> deep` 升级，避免统一扩大思考预算。
5. 建立真正的流式 TTFT、Reasoning/正文完成度和显式测试流量标签。

完整顺序、验收门槛和三层职责见
[`ENHANCEMENT_ROADMAP.md`](ENHANCEMENT_ROADMAP.md)。

### P2：按需求扩展

1. 在串行候选端口评估 Q6_K 权重，必须同时满足显存、长上下文和闭环质量门禁。
2. 独立视觉 Profile 与多模态验收。
3. 其他模型或推理后端的可替换部署档案。
4. 并发需求明确后再评估连续批处理、多 GPU、vLLM 或 SGLang。

任何自动化只能观测和验收，不能未经 A/B 自动修改 KV 精度、上下文、思考模式、
Tool Use 严格度或路由。
